# 情侣点菜 App 产品需求文档

> 版本：2.0.0  
> 更新：2026-07-03  
> 状态：以当前源码主线为准

## 1. 产品定位

这是一个 Android 情侣点菜 App。核心不是外卖平台，也不是多商家市场，而是两个人围绕“我的店铺”完成一条轻量点菜链路：

```text
登录 / 注册
  -> 情侣首页
  -> 选择当前角色
  -> 我的店铺点菜
  -> 购物车
  -> 结算
  -> 订单 / 订单详情
```

发现页只负责搜索菜品，并把合适的菜加入“我的店铺”。我的页负责账号资料、伴侣绑定、通知设置，以及进入“我的店铺”编辑。

## 2. 当前页面

### 2.1 登录 / 注册

- 登录页：邮箱和密码登录。
- 注册页：包含账号、密码、确认密码、昵称和头像资料步骤。
- 注册时两次密码必须一致。
- App 名称未最终确认前，用户界面不强行展示 `OrderDisk` 品牌文案。

### 2.2 情侣首页

- 展示当前用户头像位、中间爱心、伴侣或邀请位。
- 当前用户固定在左侧，不因角色切换改变位置。
- 当前用户头像下方展示当前角色：饲养员 / 吃货 / 选择身份。
- 未绑定伴侣时，右侧显示邀请位或空心加号。
- 选择身份只保存身份并更新页面，不直接跳转点菜页。
- 饲养员功能入口：上传菜单、管理我的店铺。
- 吃货功能入口：进入我的店铺点菜。
- 首页展示纪念日入口和一起吃饭天数。
- 有未完成订单时，首页展示订单提醒卡片：
  - 饲养员看到“有新的点菜单等你接单”。
  - 吃货看到“订单已提交，等饲养员接单”。

### 2.3 点菜页

- 页面形式可以借鉴美团点菜效率，但业务语义是“我的店铺”。
- 展示店铺信息、分类栏、菜品列表、菜品详情和购物车浮层。
- 加入购物车后可调整数量。
- 费用文案使用“服务费”，不使用外卖平台的“配送费”语义。

### 2.4 发现页

- 只做一个核心功能：搜索菜品。
- 搜索结果可以加入“我的店铺”。
- 重复加入时需要提示“已在我的小店”。
- 加入前需要确认价格和分类，避免脏数据进入菜单。

### 2.5 我的店铺

- 管理“我的店铺”的分类、菜品、价格、库存、上下架。
- 支持批量选择、批量上架、批量下架、批量删除、批量移动分类。
- 这是饲养员角色的主要工作区。

### 2.6 购物车 / 结算

- 购物车展示已选菜品、数量、小计、服务费和合计。
- 结算页收集联系人、联系电话、收餐地址和订单备注。
- 空购物车不能提交订单，并提示“购物车为空，先去点菜吧”。
- 联系人或收餐地址为空时，提示“请填写联系人和收餐地址”。
- 提交中按钮显示“提交中...”，并防止重复提交。
- 提交失败时提示“提交失败，请稍后再试”。

### 2.7 订单 / 订单详情

- 订单页展示厨房订单和美食日记两个方向。
- 当前已落地厨房订单列表、日期筛选和订单详情。
- 订单状态：
  - `submitted`：已提交
  - `confirmed`：饲养员已接单
  - `delivering`：准备中
  - `completed`：已完成
  - `cancelled`：已取消
- 订单详情支持饲养员接单、开始准备、完成这顿饭和取消订单。
- 状态变化后显示提示信息。

### 2.8 我的页

- 展示头像、昵称和个人中心入口。
- 支持设置头像和名称。
- 支持订单通知开关。
- 支持邀请小伙伴、输入 6 位绑定码、解除绑定。
- 支持进入我的店铺编辑。

## 3. 当前导航

主 Tab：

```text
HOME      -> CoupleMenuScreen
ORDERING  -> OrderingScreen
DISCOVER  -> DiscoverScreen
ORDERS    -> OrdersScreen
PROFILE   -> ProfileScreen
```

其他路由：

```text
AUTH
ONBOARDING
ANNIVERSARY
SHOP_SETTINGS / MENU_MANAGEMENT
CART
CHECKOUT
ORDER_DETAIL
```

## 4. 数据与仓储

- `SingleShopRepository`：当前“我的店铺”的单店抽象。
- `RoomMenuRepository`：维护我的店铺菜品。
- `RoomCartRepository`：维护本地购物车。
- `RoomAddressRepository`：维护本地地址。
- `SupabaseOrderRepository`：订单本地保存优先，登录后同步 Supabase。
- `SupabaseProfileRepository`：个人资料和伴侣绑定信息。
- `HybridDishRepository`：旧版菜品库兼容能力，不作为新主流程默认依据。

## 5. 设计原则

- 不要把 App 设计成多商家外卖平台。
- 不要默认存在附近商家、推荐商家、骑手配送、平台补贴。
- 不要在用户未确认 App 名称前，把 `OrderDisk` 做成用户可见主品牌。
- 视觉围绕两只小狗、情侣点菜、我的店铺、温暖点餐。
- App 图标方向：小金毛 + 马尔济斯，小金毛戴厨师帽，简约突出重点。
- 首页不展示 App 图标，首页重点是两个人、角色、纪念日和订单提醒。

## 6. 已知待补

### 高优先级

- 伴侣协作还需要更完整：对方在线、对方角色、对方是否已看到订单。
- 订单通知目前主要是 App 内提醒，后续可扩展系统通知或推送。
- 订单状态操作还需要进一步加角色权限，例如只有饲养员可以接单和开始准备。
- 美食日记还需要真正接入已完成订单。

### 中优先级

- 我的店铺需要更完整的图片、口味、标签和库存编辑体验。
- 发现页加入菜品时可增加去重、分类推荐和价格建议。
- 结算页可增加常用地址、默认联系人和备注模板。
- 订单详情可增加时间线和更明确的伴侣状态反馈。

### 低优先级

- 兼容模块中的随机推荐、心愿单、餐次等能力需要决定保留、移除或整合。
- 旧文档和旧截图需要继续清理，避免误导后续设计。
- Figma 页面需要继续与当前源码结构保持同步。

## 7. 不再作为当前主线的方向

除非重新确认产品方向，否则以下内容不作为当前开发和设计依据：

- 多商家外卖市场
- 附近商家 / 推荐商家 / 商家排行榜
- 骑手配送
- 平台补贴、商家经营后台、外卖履约后台
- 旧版随机推荐作为主首页
- 旧版心愿单作为主流程
- 旧版餐次记录作为主流程

## 8. 验收标准

- 登录 / 注册、情侣首页、角色切换、纪念日、点菜、发现搜菜、加入我的店铺、我的店铺编辑、购物车、结算、订单列表、订单详情均可在当前主线中连通。
- 首页只保留一套情侣首页结构，不出现新旧两套 Tab。
- 角色切换只更新当前用户角色文本，不直接跳转点菜页。
- 当前用户始终显示在左侧，伴侣或邀请位显示在右侧。
- 用户界面不出现乱码，不出现旧外卖平台语义。
- Debug 包可成功编译。
